最近在整理自己的技术笔记，顺便也准备把这个账号重新捡起来。

点进我主页的人可能会有点困惑：以前还在写前端、JavaScript、页面性能优化，怎么突然开始写 ARMv8、TrustZone、Secure Boot 和 TEE 了。

账号没卖，也没被盗。

只是这些年工作内容变了。

我最开始做的是应用开发，主要偏前端。那个阶段关心的问题，大多是页面、接口、状态管理、性能和工程化。

后来进入智能座舱领域，开始接触平台安全，工作内容也慢慢往 SoC Security、Secure Boot、TEE、密钥管理、Linux Kernel 和 Android 平台安全这些方向靠。

以前排查问题，经常从浏览器控制台开始。

现在排查问题，有时要从芯片上电开始。

跨度确实有点大。

但做了一段时间之后，我反而发现，这两个阶段并没有想象中那么割裂。以前关心的是一个功能为什么没有正常工作，现在关心的是一段代码为什么有资格被执行、一份数据为什么可以被访问、一个设备为什么能够被信任。

问题变底层了，但“这玩意为什么不按预期运行”始终是工程师绕不过去的主线。

为什么突然想整理知识体系

做平台安全之后，我陆续接触了不少东西。

比如 Secure Boot、Chain of Trust、TrustZone、TEE、TA、CA、SMC、TF-A、OTP、eFuse、RPMB、HUK、KeyMint、AVB、anti-rollback。

这些词单独拿出来，我多少都能讲一点。

但有一段时间，我越来越明显地感觉到：自己知道的东西不少，但这些东西在脑子里没有真正连起来。

今天看 Secure Boot，知道镜像要验签。

明天看 TrustZone，知道系统分成 Secure World 和 Normal World。

后天看 TEE，知道 CA 通过驱动和 TA 通信。

再过两天看 TF-A，开始研究 BL1、BL2、BL31、BL32、BL33。

每一块单独看，好像都懂。

但如果有人突然问：

“它们到底是怎么串起来的？”

我可能需要先沉默几秒，然后在脑子里临时画一张图。

这也是我现在最想解决的问题。

我不想继续以“遇到一个词，搜索一个词，收藏十篇文章”的方式学习了。

这种学习方式短期很有满足感。浏览器标签页越开越多，仿佛知识也在同步增长。

实际上增长的可能只是内存占用。

真正需要建立的，不是一堆彼此独立的知识点，而是一张完整的系统地图。

我需要知道每个模块为什么存在、解决什么问题、依赖谁、信任谁，以及出了问题之后会影响哪一层。

所以我准备重新整理自己的 SoC Security Knowledge Base。

这件事一方面是为了工作和学习，另一方面也是为了以后能够持续输出一些真正成体系的内容，而不是今天写 TEE，明天写 Kernel，后天突然心血来潮开始讲证书链，文章之间互相不认识。

我准备围绕八个方向整理

目前我把这套知识体系分成八个部分。

它不一定是最终版本，后续肯定还会调整。但至少现在，它可以先作为一张地图。

1. SoC Security Architecture

这一部分先不钻细节，主要回答一个问题：

一颗 SoC 的安全能力，整体上到底是怎么组织起来的？

我会从 Root of Trust、硬件隔离、启动信任链、可信执行环境、密钥体系、操作系统安全这些角度，梳理各个模块之间的关系。

这一层很重要。

如果连全局架构都没看清楚，就直接开始研究某个寄存器或者某段源码，很容易学着学着迷路。

我以前也干过这种事。

一个细节研究了两天，最后突然发现：好像并不知道它在整个系统里负责什么。

2. ARMv8 / TrustZone / Exception Model

这一部分主要整理 ARMv8 里的异常级、安全状态、内存与权限模型。

包括 EL0 到 EL3 分别做什么，Secure World 和 Normal World 是怎么划分的，SMC 是怎么进入 EL3 的，以及异常、中断、上下文切换和内存访问控制之间的关系。

TrustZone 很容易被讲成一个神秘的“安全世界”。

但继续往下看就会发现，它并不神秘。

它依然要处理寄存器、页表、异常、中断和内存。

安全世界不是魔法世界，只是权限更高，责任也更大。

写错代码的时候，崩得同样很真实。

3. Secure Boot / Chain of Trust / TF-A

这一部分会从芯片上电开始，梳理一条完整的启动信任链。

比如 Boot ROM 为什么可以作为起点，Root Public Key Hash 如何和 OTP 或 eFuse 关联，镜像怎样签名和验签，TF-A 中的各个启动阶段分别负责什么，信任如何继续传递到 UEFI、U-Boot、Linux Kernel 和 Android。

还会包括 anti-rollback。

毕竟“这个固件是官方签名的”，不代表“这个固件就一定应该被允许启动”。

十年前的官方漏洞版本，依然可以是官方的。

我的账号名 RootPKhash，也来自这里。

它本质上只是一个公钥哈希。

但沿着它往后走，会牵出密钥、芯片、启动、生产和设备身份一整套东西。

作为网名确实不像正常人会起的名字，不过工程师的网名，能读出来已经算比较友好了。

4. TEE / TA / CA / SMC

这一部分会整理 TEE 的完整调用链。

从普通世界里的 CA，到 Kernel TEE Driver，再到 Secure Monitor、TEE OS 和 TA，看看一次请求到底是怎么进入安全世界，又是怎么返回结果的。

也会关注共享内存、Session、Command、RPC、权限校验、TA 生命周期和安全存储这些问题。

TEE 很容易给人一种错觉：代码只要放进 Secure World，就安全了。

其实并不是。

代码写得不安全，放进 Secure World 之后，只是变成了一个权限更高的漏洞。

5. Linux Kernel / BSP / Driver / Device Tree

这一部分是我后面想重点补强的方向之一。

很多 SoC 安全能力最后都需要通过 Kernel、Driver 和硬件真正连起来。

Device Tree 怎么描述设备，驱动怎么访问寄存器，Kernel 怎么和 TEE 通信，用户态请求怎么经过 ioctl、共享内存和设备节点一路走到底层，这些都需要理解。

安全架构图里的箭头通常画得很简单。

真正开始实现之后，才会发现每一根箭头里面都塞了很多东西。

而很多问题，恰恰就藏在箭头里。

6. Key Management / HUK / RPMB / Secure Storage / KeyMint

这一部分主要研究密钥和敏感数据。

密钥从哪里产生，怎么派生，存在哪里，谁能使用，什么时候失效，设备恢复出厂之后怎么办，软件升级之后怎么办。

还会整理 HUK、RPMB、安全存储、KeyMint 和设备身份之间的关系。

密码算法本身通常有成熟标准。

工程中更容易出问题的，反而是密钥生命周期。

比如密钥被写进日志，测试密钥忘记替换，权限控制过宽，或者某个“临时方案”活了三年。

所有长期方案，往往都曾经是临时方案。

7. Android / Automotive Platform Security

这一部分会继续往上走，整理 Android 和智能座舱平台里的安全机制。

包括 AVB、SELinux、Binder、Keystore、KeyMint、HAL、系统服务、OTA 和回滚保护，也包括这些上层机制如何调用底层 SoC 的安全能力。

SoC Security 并不是 Kernel 启动成功之后就结束了。

信任链还要继续往上走。

越往上，组件越多，业务越复杂，开发人员也越多。

系统开始逐渐进入大家熟悉的状态：理论上有清晰边界，现实中有大量历史原因。

8. Security Engineering / Threat Modeling / Production Security

最后一部分不局限于某一个技术模块。

我想整理的是，安全能力如何从设计真正走到产品里。

比如怎样做威胁建模，怎样识别资产和攻击面，怎样定义信任边界，怎样设计安全需求，以及生产阶段的密钥注入、证书、设备授权、版本管理、维修和漏洞响应。

做安全不能只盯着代码。

一套方案即使设计得很好，如果生产流程、升级流程或者售后流程没有跟上，最后仍然可能出问题。

安全不是一个单独的模块。

它更像是散落在整条链路里的约束条件。

只是平时不一定有人注意，直到某个条件被违反。

后续文章准备怎么写

以后输出这些内容时，我会尽量避免两种情况。

一种是只讲概念。

看完之后知道“Secure Boot 很重要”“TEE 很安全”“HUK 不能泄露”，但具体为什么、怎么做、数据怎么走，还是不知道。

另一种是直接扎进源码。

上来就是函数、结构体、宏定义和调用栈，读了半天，连当前在解决什么问题都忘了。

我更希望按照一个相对固定的顺序去写：

先讲整体架构，再讲设计目的和模块职责，然后梳理关键数据流，最后再进入代码、风险点和工程实践。

也就是说，先回答：

- 这是什么；
- 为什么需要它；
- 它在整个系统里的位置是什么；
- 它和其他模块怎么配合；
- 一次真实请求是怎么流转的；
- 如果设计错了，会发生什么。

之后再谈源码。

我自己学习的时候，也会按照这个顺序来整理。

毕竟源码不会跑。

但人在里面很容易跑丢。

关于资料来源和边界

这个系列只会基于公开资料进行整理。

主要包括 ARM 官方文档、Trusted Firmware-A、OP-TEE、Linux Kernel、Android Security、GlobalPlatform 规范、芯片厂商公开手册，以及公开论文和安全研究资料。

我会结合自己的工作理解，尽量把这些资料重新组织成更容易理解的结构。

但不会涉及公司内部代码、内部架构、项目细节、客户信息、未公开方案或者具体实现。

这里需要把边界讲清楚。

工作经历会影响我怎么理解一个问题，但不代表我会把工作内容直接写出来。

简单来说：

我会写自己基于公开资料理解了什么，不会写公司内部具体是怎么做的。

一个写安全的账号，第一篇转型文章就先把安全红线踩了，多少有点不合适。

以前的前端文章会不会删

不会。

那些文章虽然和现在的方向不完全一样，但确实是我过去经历的一部分。

从应用开发到平台安全，也不是彻底推翻过去重新开始。

相反，应用开发经历让我更容易理解：底层安全能力最终是如何被上层使用的，以及上层使用不当时，底层设计得再完善也不一定有用。

以前我关心的是页面显示了什么。

现在我更关心，页面亮起来之前，系统已经做了哪些决定。

所以历史前端文章会保留。

它们可以解释我从哪里来。

之后的内容，则会慢慢说明我准备往哪里走。

写在最后

这篇文章算是这个账号的新起点，也是后续 SoC Security 系列的总入口。

不保证更新速度有多快，也不打算把自己包装成什么安全专家。

更准确地说，我只是想把工作中接触到的问题、公开资料里学到的知识，以及自己反复想不明白的地方，重新整理成一套能够长期使用的知识体系。

后来发现，这件事和“等需求完全明确了再开始开发”差不多。

一般等不到那一天。
