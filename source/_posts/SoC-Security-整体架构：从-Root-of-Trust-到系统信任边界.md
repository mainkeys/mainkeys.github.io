这是 SoC Security 系列里真正开始讲技术结构的第一篇。

如果说上一篇更像是在回答“我为什么开始系统整理这件事”，那么这一篇想回答的问题是：

一套 SoC Security，到底由哪些部分构成，它们之间又是怎么连起来的？

很多人第一次接触平台安全时，通常会先被一堆名词砸中：

Boot ROM、eFuse、OTP、Root Public Key Hash、Chain of Trust、TrustZone、TEE、TA、RPMB、HUK、AVB、KeyMint。

这些词单独看都不难。

真正难的是，当它们同时出现在一个系统里时，你能不能说清楚：

- 谁是信任起点；
- 谁负责隔离；
- 谁负责验签；
- 谁负责保管密钥；
- 谁负责把底层安全能力真正暴露给操作系统和应用；
- 一旦某一层失守，会向上或向下影响到哪里。

所以这一篇先不急着钻某个寄存器、某个源码目录或者某条 SMC 调用链。

我们先把地图画出来。

## 为什么一定要先看整体架构

如果没有整体架构，很多学习会很快变成“局部懂了，整体还是糊的”。

今天看 Secure Boot，知道镜像要验签。

明天看 TrustZone，知道 Secure World 比 Normal World 权限高。

后天看 TEE，知道 CA 会通过驱动和 TA 通信。

但这些知识如果没有被放进同一张图里，就很容易变成分散的记忆点。

而 SoC Security 恰恰不是一个单点技术。

它更像是一整套协作系统：

- 硬件提供最底层的不可篡改能力；
- 启动链决定“谁有资格运行”；
- 隔离机制决定“谁能碰到谁”；
- 密钥体系决定“谁可以解密、验签、证明身份”；
- 操作系统和平台框架决定“这些能力最后如何被上层调用”；
- 生产与运维流程决定“设计好的安全机制能不能活到真实产品里”。

所以看整体架构，不是为了追求抽象。

恰恰相反，是为了以后在看细节时，不会把自己看丢。

## 一张简化后的 SoC Security 地图

如果把 SoC Security 粗略分层，我会先把它拆成下面六层。

### 1. Immutable Root 层

这是整条信任链最靠下的一层。

典型元素包括：

- Boot ROM；
- OTP / eFuse；
- Root Public Key Hash；
- 硬件随机数源；
- 硬件密码模块；
- Debug 生命周期控制。

这一层最核心的价值，是提供一个足够早、足够稳定、足够难被修改的信任起点。

系统要想验证后续镜像是否合法，必须先有一个“默认可信”的东西。这个东西通常不可能来自可写存储本身，否则信任会陷入循环。

所以很多平台都会把最早的启动代码放在 Boot ROM，把公钥哈希或者设备身份相关材料放在 OTP 或 eFuse 一类的不可轻易修改区域。

### 2. Isolation & Privilege 层

这一层主要回答两个问题：

- 代码运行在什么权限级；
- 不同世界、不同地址空间、不同外设之间如何隔离。

这里常见的组成包括：

- ARMv8 Exception Level；
- TrustZone Secure / Non-secure 状态；
- MMU 与页表权限；
- 中断路由；
- 外设访问控制器；
- 内存保护与总线访问控制。

很多人会把 TrustZone 理解成“安全功能本身”。

我更倾向把它理解成“安全功能得以存在的隔离基础设施”。

它不直接等于安全，但没有它，很多安全能力很难稳妥落地。

### 3. Boot & Verification 层

这一层决定的是：设备从上电到系统启动，信任是怎么一跳一跳往上传的。

典型链路通常类似：

`Boot ROM -> BL1 / BL2 -> BL31 -> BL32 / BL33 -> Bootloader -> Kernel / Android`

具体阶段名称因平台而异，但核心逻辑基本一致：

- 上一个阶段验证下一个阶段；
- 只有验证通过，控制权才继续向后传递；
- 每一跳都要明确“验证谁、用什么验证、失败时怎么办”。

这就是大家常说的 Chain of Trust。

这里真正重要的，不只是“做了验签”，而是整条链不能断、不能绕、不能被旧版本回滚利用。

### 4. Trusted Runtime & Service 层

设备启动起来之后，安全并没有结束。

接下来系统还需要一块受保护的运行环境，去承载高敏感度的安全服务，比如：

- TEE OS；
- Trusted Application；
- Secure Monitor 协调；
- 安全存储；
- 密钥派生与使用；
- 安全计数器、认证服务、证明服务。

这一层的价值，在于给“启动之后仍然需要被保护的能力”提供一个长期驻留的可信执行环境。

否则，系统即使启动时是可信的，运行时也很可能被高权限软件逐步侵蚀。

### 5. OS & Platform Integration 层

这一层很容易被低估，但它实际上决定了底层安全能力能不能被真正用起来。

它通常包括：

- Linux Kernel；
- TEE Driver；
- BSP 与设备驱动；
- Android HAL / Framework；
- KeyMint / Keystore；
- AVB、SELinux、OTA、Rollback Protection。

从工程视角看，很多安全能力不是“设计完就存在”，而是要经过驱动、设备节点、共享内存、RPC、系统服务、权限控制，一层一层接出去。

安全架构图里的箭头看起来都很短。

真正实现时，每一根箭头都可能对应几千行代码和一堆边界条件。

### 6. Provisioning & Lifecycle 层

最后一层往往不是最“技术宅”的，但对真实产品非常关键。

这里关注的是：

- 密钥如何注入；
- 证书如何烧录；
- 设备身份如何绑定；
- 版本如何管理；
- 调试口如何关闭或受控开启；
- 返修、翻新、退役、漏洞响应怎么处理。

一套安全方案如果只在实验室里成立，而进到量产、升级、售后之后漏洞百出，那它就不能算完整。

安全设计必须贯穿产品生命周期，而不是停在架构图。

## 这六层之间到底是什么关系

如果只记一句话，我会记这个：

**SoC Security 不是若干安全模块的堆叠，而是一条从硬件信任根出发、经过隔离和验证、最终延伸到运行时与产品生命周期的连续链路。**

更直白一点说：

- Immutable Root 提供起点；
- Isolation 负责边界；
- Boot Chain 负责准入；
- Trusted Runtime 负责高敏感操作；
- OS Integration 负责把能力接出去；
- Lifecycle 负责让它在真实世界里不失效。

六层里任何一层出问题，都可能让前面的投入被后面绕开。

比如：

- 启动链很强，但量产时测试密钥没换掉；
- TEE 很强，但普通世界驱动权限校验太松；
- 密钥派生设计很强，但日志把敏感材料打出来了；
- Secure Boot 很强，但回滚保护没接好，旧漏洞版本照样能跑。

所以判断一套平台是否安全，不能只看单个模块是否“存在”，而要看整条链是否真正闭合。

## 一个最常见的完整数据流

为了把这些层串起来，我们可以看一个简化版本的真实流程：

### 场景：设备上电后，应用请求一个受保护的密钥服务

1. 芯片上电，Boot ROM 开始执行。
2. Boot ROM 从 OTP / eFuse 里拿到预置的可信锚点，验证下一阶段镜像。
3. 后续 Bootloader / TF-A 阶段继续把信任传递到 Secure World 和 Normal World 的运行环境。
4. TEE OS 被加载并进入可服务状态，安全存储和密钥服务可用。
5. Linux / Android 启动后，通过驱动和系统服务把部分安全能力暴露给上层。
6. 应用层通过 Framework 或系统 API 发起请求，最终进入 KeyMint、TEE Driver、SMC、TEE OS，再抵达具体 TA。
7. TA 在受保护环境里使用密钥或完成校验，只把必要结果返回普通世界。

看起来像一个“应用调用接口”的过程，实际上中间穿过了：

- 启动信任链；
- 安全世界切换；
- 权限边界；
- 共享内存与驱动路径；
- 密钥生命周期管理；
- 平台框架封装。

所以平台安全最有意思的地方就在这里：

很多问题表面看是“上层接口不对”，根因却可能埋在启动链、隔离配置、驱动设计或者量产流程里。

## 我现在最在意的几个判断标准

如果以后我要快速评估一套 SoC Security 设计，我大概率会先问下面这些问题：

- 信任根放在哪里，是否真的不可被后写入内容替代；
- 启动链是否逐级验证，失败路径是否清晰；
- Secure / Non-secure 边界是否明确，是否有共享资源被滥用；
- TEE 中到底承载了哪些真正高价值能力；
- 密钥是如何生成、派生、存储、销毁和迁移的；
- OS 和驱动层是否把底层安全能力错误地暴露过宽；
- 回滚、维修、调试、量产这些生命周期场景是否被设计进去。

这几个问题未必能一次性回答全部细节，但足够帮助我判断：这套架构到底是在“认真做系统安全”，还是只是把安全名词都挂上去了。
